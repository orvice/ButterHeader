# 0003. Config Store 统一配置读写，冲突时未保存的本地编辑优先

## 状态

已接受（2026-07-07）

## 背景

配置的持久化约束（ADR 0002：独立 key 布局、防抖写入、配额报错）此前分散在 `storage.ts`、`save-queue.ts` 和 Options 页组件内的胶水代码中。最微妙的逻辑——外部变更（Popup、其他设备）到达时如何不覆盖仍在防抖窗口内的本地编辑——依赖 `document.hasFocus()` 启发式，无法测试。

## 决策

引入 **Config Store**（`src/core/config-store.ts`）作为各界面读写配置的唯一入口：

- Store 持有内存中的 Config 快照作为单一事实来源；修改同步生效（乐观更新），随后写回存储。
- 存储通过 **chrome 形状的 KV 适配器**（`get/set/remove/onChanged`）注入；ADR 0002 的 key 布局成为 Store 的私有实现。测试使用内存假适配器。
- 只有 `updateProfile`（打字高频）走防抖；增删、排序、全局暂停、导入均立即批量写入（导入为单次原子 set）。
- **协调策略：外部变更按 key 应用，但处于待保存状态的 Profile 保留本地值**（随后落盘胜出）。自身写入的回声通过值相等判定自然消化。
- 保存失败进入可订阅状态 `{ config, saveError }`，随成功写入清除；编辑内容永不静默丢失。

## 取舍与后果

- ✅ 编辑覆盖竞态从 UI 启发式变为可用假适配器 + 假定时器单测的确定性行为。
- ✅ 三个界面（Options、Popup、后台）成为 `subscribe` 之上的薄适配器，语义一致。
- ❌ 跨设备真冲突按 Profile 粒度 last-write-wins——与 chrome.storage.sync 本身语义一致，接受。
- ❌ 内存快照与存储存在短暂分歧窗口（防抖期间）。这是有意的：等待落盘的替代方案（外部变更总是获胜 / 先强制刷盘再应用）要么重新引入丢键覆盖，要么让每次外部事件消耗写配额。

## 备选方案

- **外部变更总是获胜**：重新引入本插件要修复的编辑覆盖 bug，排除。
- **先刷盘再应用外部变更**：结果依赖事件时序、不可预测，且浪费写频率配额，排除。
